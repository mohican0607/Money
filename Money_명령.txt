* RAG 결과를 토대로 LLM을 호출하는 방식
 1. 사용자 질문           : 사용자가 질문을 던지면,
 2. 데이터 검색(Retrieval): LLM이 바로 답하는 것이 아니라, 현대카드가 구축한 **'유니버스(태그 데이터베이스)'**에서 질문과 관련된 고객 성향, 결제 데이터, 가맹점 정보를 먼저 찾아냅니다.
 3. 프롬프트 보강(Augmentation): 검색된 정교한 데이터(태그)를 질문과 합칩니다.
 4. LLM 생성(Generation) : 보강된 정보를 GPT-4와 같은 LLM에 전달하여 최종 답변을 생성합니다.

2. 현대카드 RAG의 독특한 구성 방식: "Structured RAG"
   현대카드는 일반적인 텍스트 위주의 RAG와 차별화된 **'구조화된 데이터 기반 RAG'**를 구성합니다. (근거: 현대카드 유니버스 수출 관련 보도자료 및 기술 소개)
  * 데이터 큐레이션 (Tagging): 유니버스의 핵심은 2,600억 건 이상의 결제 데이터를 AI가 이해하기 쉬운 **'태그(Tag)'**로 미리 변환해 둔 것입니다.
  * 벡터 검색 + 태그 매칭: 사용자의 의도를 분석한 뒤, Vector DB에서 유사한 텍스트를 찾는 것에 그치지 않고, 해당 고객의 **소비 태그(취향, 라이프스타일)**를 결합하여 검색 결과의 정확도를 극대화합니다.
  * 비식별화 레이어: 금융 보안을 위해 고객의 민감한 원본 데이터는 내부 시스템에 머물고, **비식별 처리된 분석 정보(태그)**만 LLM에 전달되는 보안 아키텍처를 갖추고 있습니다.

* RAG
호출 형태: "API-First & Metadata Injection"
	현대카드는 LLM을 직접 호출하기 전, 중간에 **오케스트레이터(Orchestrator)**라는 제어 시스템을 둡니다.
  * 함수 호출(Function Calling) 방식   : 사용자가 "이번 달 내 소비 패턴에 맞는 카드를 추천해줘"라고 하면, LLM이 직접 답하는 것이 아니라 **'고객 소비 태그 조회'**라는 내부 API(함수)를 먼저 호출합니다.
  * 메타데이터 주입(Metadata Injection): API를 통해 유니버스 DB에서 추출된 '태그' 데이터(예: 주유 30%, 온라인 쇼핑 40%, 선호 브랜드: 현대백화점 등)가 JSON 형태로 추출됩니다. 이 정형 데이터가 LLM의 프롬프트 컨텍스트에 '주입'되는 형태로 호출이 이루어집니다.
  * 보안 게이트웨이                    : 외부 LLM(GPT-4 등)으로 데이터를 보낼 때, 개인 식별 정보(PII)를 자동으로 마스킹하거나 토큰화하여 보안을 유지하는 전용 게이트웨이를 거쳐 호출합니다.
